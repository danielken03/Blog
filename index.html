Functional Code VS. Scalable Code

How do you structure your code?

Friday, April 17th 2026

Programming is hard,

I have seen first hand that this saying is true, "The more I learn the less I know." As I am learning more and more about programming, it is difficult for me to be confident about anything I've learned. Something I have been learning about the past couple weeks is programming paradigms, specifically Object Oriented Programming. "a model for something that may be copied, especially : an exceptional example or archetype." - Merriam Webster. Simply put, a paradigm is a perfect example of how to structure something. In programming you need a structure for your project, you could just make a massive list of objects and functions that are all sequenced in the order they are used, but this comes with a lot of problems. The biggest problem is when you need to make changes or add/integrate parts of your code. In this blog, I am going to break down what Object Oriented Programming is, what I've learned over the last couple weeks, and how I have integrated it into my own project.

What's the secret recipe?

A great way to describe the common mistake with vibe coding, or code written by a newbie, is to describe it like a cooking recipe. I have made some pretty large coding projects, with the help of AI of course, but I have been writing the worst recipe of all time that is impossible to read, except to the person who wrote it (me). flour, sugar, baking powder, salt, cocoa powder, cinnamon, ground coffee, egg, milk

With the code I have been writing, I have been writing every feature individually, and not integrating anything together. I haven't been using classes correctly. Nothing has been working together, and that's where structure is needed. Recently I haven't been actually coding a lot, but I have been reading a lot and trying to deepen my understanding of Object Oriented Programming. Let me explain what I have learned so far.

What is OOP?

Object Oriented Programming, or OOP for short, is a way to write code that organises around classes of objects rather than just actions. A class is a template that defines what an object is and what rules it follows, for example a class could be "car" and it is defined as a vehicle with 4 wheels, doors, an engine, and drives. An object is a specific instance that is created from the class. A Honda Civic would be an object, specifically it is a sporty car with a gas engine and they are front wheel drive. A Tesla is another separate object that is in the "car" class, and it features falcon doors, an electric engine, and are rear wheel drive.

How is this distinction useful? I can call  charge_battery()  for the Tesla only, because you can't charge a Civic. I can call  refill_tank()  for the Civic, because the Tesla doesn't have a gas tank. These can be triggered inside the objects themselves, so when I call these methods, I don't have to specify what car needs to have their gas refilled. Like in my recipe, I don't have to spell out every ingredient I am mixing, I can say "mix the dry ingredients with the wet ingredients." Or with the car analogy, I don't have to spell out every type of electric vehicle to be charged, and I don't have to specify that the gas vehicles do not need to be charged.

Having these objects in a class is very helpful because I can call the entire class as well. A class method like  calculate_tax()  will figure out the tax for any car, regardless of the brand.

The Four Pillars

If you have heard of OOP you may have heard of the four pillars. I will go into each of these in a bit more detail, but what are these pillars? These are our organisation tools that help us know how to write code in a way that is more simple, flexible, reusable, and protected.

Encapsulation

The first pillar is Encapsulation; the tool all about data protection. Think of a bank: if I have a bank account, I don't want any random part of code to edit my balance. I only want my balance to change when I deposit or withdraw something, which we can call with a method like deposit() or withdraw()

class BankAccount:
    def __init__(self, initial_balance):
        self.__balance = initial_balance

    def deposit(self, amount):
        if amount > 0:
            self.__balance += amount

    def withdraw(self, amount):
        if 0 < amount <= self.__balance:
            self.__balance -= amount

    def get_balance(self):
        return self.__balance

In this block of code, we are initialising  __balance  and the reference self. When I run get_balance(self) I am getting my (self) balance, and my bank account's balance is private because of the 2 underscores (__). Now, because I have set up deposit() and withdraw() inside the BankAccount class and made it private, it will only deposit inside my bank account for the amount of money I choose to deposit. Other parts of code cannot edit my balance outside of this class.

When I made this code for a bank account, I knew I needed to keep the data secure so I placed this crucial part in a class and made it private by following Encapsulation, which is using access modifiers to make things public or private inside the class.

Abstraction

The next pillar is Abstraction, or the simplifier. Abstraction is all about simplifying the code by allowing the user to focus on what the code does, rather than how the code does it. This is all about front end programming, which is what the user sees rather than what the developer sees. For example, this blog website you are on right now have some stuff in the background you don't see, for example when you click the tabs in the sidebar to switch from one post to another, you don't see how the website takes you to the post. This is pretty simple to explain, and pretty simple to execute, as it's just front end programming so I won't go into too much detail about this. Maybe someday I can post about HTML and CSS and how they work and how to make things look nice and functional, but for now let's skip it.

Inheritance

The third pillar is Inheritance, or the reusability of a class. If I want to add a new feature to my website that uses features in a class I have already created, Inheritance helps me to create a new "child" class of a "parent" class. Here is an example of a child that builds on the parent I used in the previous example:

class SavingsAccount(BankAccount):
    def __init__(self, initial_balance, interest_rate):
        super().__init__(initial_balance)
        self.interest_rate = interest_rate

    def apply_interest(self):
        current_balance = self.get_balance()
        interest_amount = current_balance * self.interest_rate
        
        self.deposit(interest_amount)

This class is a child of the parent class BankAccount, and using methods in python like super() we can allow the child to access certain parts of the parent. Since we have the parent class __balance and we have already made rules on depositing and withdrawing funds, we can give this new class access to these already set rules. The child is saying "hey dad, give me a balance that is set up with your rules please!" This is Inheritance! The SavingsAccount is a type of bank account, so it follows the same rules, but it is giving the account interest by getting the balance amount from the parent class with self.get_balance() and multiplying it by the interest rate.

Inheritance is super helpful to any project because we now don't need to recreate a whole new class or repeat old instructions; we can simplify our code and keep it organised by reusing what we have already created.

Polymorphism

Polymorphism is the final pillar, and I think is the most confusing. I still don't fully understand it, so I will let someone smarter than me explain it. Introducing a Redditor, Joewoof!

It’s a very fancy term for a very simple concept: using the same method & properties for subclasses of the same superclass. For example, all characters can attack() in a game, but how they attack is different. All characters have health, but how much is different.

The superclass might be Enemy. The subclasses might be Orc and Slime. Orc has 10 health and attacks by swinging a weapon. Slime has 5 health and attacks by spitting acid.

There’s more, but that’s the gist of it. It’s meant to simplify your code and reduce problems.
